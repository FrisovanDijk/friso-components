Friso's Component API Docs

Why a component API?

I enjoy using new technologies with every project, but that renders a lot of front-end work useless because it's heavily embedded in different templating languages and CMSes. The rest takes time and effort to reuse, because it needs to be refactored. The workflow works bad coding practices in hand.

My answer to this problem consists of 3 parts:

  1. Use Atomic Design as a philosophy for reusable components.
  2. Extend Atomic Design with Tachyons as particles.
  3. Serve the style guide through an API.

Don't worry if you don't know some of these terms. You can read more about this architecture here.

The workflow of this API is very simple: you make a request with the component name and JSON to receive a semantic component. So why do I think this design philosophy is a good idea?

  1. Agnostic. Use any framework or language you want. This API implementation uses SASS and Mustache as PHP templating library, but those are arbitrary dependencies. You can use whatever you want in your projects.
  2. Reusable. All components come from a single source. It's easy to grab your best semantic, accessible HTML. Class names are the same across projects.
  3. Documentable. Document new components with a single Javascript function call.
  4. Discoverable. Browse the documentation with examples of components to see what would work for you.
  5. Usable. A CSS call and an API wrapper is all you need to get started. The Jacascript wrapper for the documentation is a whopping 36 lines.
  6. Stateless. Separate concerns by using data to generate components. No mixing of HTML and data involved.

Take a look around. The project is open source and available on Github, or get inspired and implement your own.

Using the Component API

Using it in a project is extremely simple. This documentation site uses a very light Javascript implementation (although it doesn't support IE).

If you use it, make sure to set var apiUrl on line 4 to your API Url.

To render a component with it, simply use the Javascript code below:

parse_componentHTML(identifier, {
"name": component name,
"vars": {
    component variables
}
});

Ok, so what can I input? The identifier is your in-document identifier and uses document.querySelector(). The second argument is the JSON to generate the component. The name is the type of component you generate, and vars is the data for your component.

Switching the branding colors is easy to do. Open source/css/core-css/_variables.scss and change the big map of colors. Compile the stylesheet and you're good to go (or check out Koala if you don't know how)
